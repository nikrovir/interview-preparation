# Git
> **Git** — это распределённая система управления версиями, используемая для отслеживания изменений в файлах и координации работы над проектом среди нескольких людей.

### Как создать новый репозиторий в Git?
Для создания нового репозитория используйте команду ```git init``` в директории проекта. Это создаст поддиректорию .git, которая содержит все файлы, необходимые для репозитория.

### Как клонировать существующий репозиторий?
Используйте команду ```git clone <url>```, где <url> — это URL репозитория. Это создаст локальную копию репозитория.

### Как создать новую ветку и переключиться на неё?
Чтобы создать новую ветку, используйте ```git branch <branch_name>```. Чтобы переключиться на эту ветку, используйте ```git checkout <branch_name>```. Совместная команда: ```git checkout -b <branch_name>```.

### Как удалить ветку?
Для удаления локальной ветки используйте ```git branch -d <branch_name>```. Для принудительного удаления ветки используйте ```git branch -D <branch_name>```. Для удаления удаленной ветки используйте ```git push origin --delete <branch_name>```.

### Как просмотреть историю коммитов?
Используйте команду ```git log```. Дополнительные параметры, такие как --oneline, --graph, могут быть использованы для более удобного просмотра.

### Как отменить изменения в файле, если они ещё не были закоммичены?
Используйте команду ```git checkout -- <file>```, чтобы вернуть файл к состоянию последнего коммита. Если изменения были добавлены в индекс, используйте ```git reset HEAD <file>``` перед этим.

### Как отменить последний коммит?
Используйте команду ```git reset --hard HEAD~1```, чтобы удалить последний коммит и изменения в рабочей директории. Для сохранения изменений используйте ```git reset --soft HEAD~1```.

### Как добавить удалённый репозиторий?
Используйте команду ```git remote add <name> <url>```. Обычно <name> — это origin.

### Как отправить изменения в удалённый репозиторий?
Используйте команду ```git push <remote> <branch>```, например, git push origin master.

### Как разрешить конфликт слияния?
Откройте конфликтующие файлы, найдите маркеры конфликтов (строки, начинающиеся с <<<<<<<, =======, >>>>>>>), разрешите конфликт вручную, затем добавьте исправленные файлы в индекс и завершите слияние с git commit.

### Что такое rebase и как он отличается от merge?
```git rebase``` позволяет интегрировать изменения из одной ветки в другую, перемещая или "перезаписывая" ваши локальные коммиты на вершину другой ветки (создавая линейную историю коммитов). В отличие от merge, который сохраняет историю всех коммитов, rebase делает историю более линейной и чистой.

> Основная фишка rebase он историю твоих коммитов кладет поверх той ветки, куда делается rebase, из-за чего у всех твоих коммитов изменится хэш, дата коммита останется старой, но дата правок будет текущей. Преимущество такого подхода, то что master может смерджится с этой веткой без merge коммита, через fast-forward и риск получения конфликтов минимален.

**Примеры с коммитами:**

Предположим, у вас есть две ветки: master и feature. Вы внесли изменения в ветку feature и хотите объединить эти изменения с веткой master.

```
master: A --- B --- C
                     \
feature:              D --- E
```

Результат после слияния (merge):
```
master: A --- B --- C --- F
                    \     /
                    D --- E
```

Результат после слияния (rebase):
```
master: A --- B --- C
                     \
feature:              D' --- E'
```

Т.е. при rebase все изменения накладываются поверх имеющихся (перезаписываются), а при merge создаётся новый коммит и происходит сохранение всей истории коммитов.
